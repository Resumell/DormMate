## 数据流

### 1. 传感器上报流（核心循环）

```
ESP32 sensor_task (每10秒)
  │
  ├── 采集: DHT11温度/湿度 + 光线ADC + PIR红外 + 电流
  │
  ├── POST /api/report { sensors, context }
  │     │
  │     ├── 更新 latest_sensors 缓存
  │     ├── context.event == "human_first"
  │     │     └── 写入 return_history 表
  │     ├── context.query_type == "predict"
  │     │     └── ask_ai() → 预测模式 → 返回 preheat/voice_notify=10
  │     ├── 普通上报
  │     │     └── ask_ai() → 环境异常检测 → 返回 voice_notify=1/4
  │     └── 返回 JSON { actions: { relay, voice_notify, tts, oled } }
  │
  └── ESP32 解析 actions → 执行继电器 + 语音播报 + OLED更新
```

### 2. 语音播报流（优先级队列）

```
后端返回 actions.voice_notify
  │
  ├── voice_notify == 0  → 不播报
  ├── voice_notify == 1  → 插入语音队列("是否开灯")
  ├── voice_notify == 4  → 插入语音队列("是否开空调")
  ├── voice_notify == 9  → 插入语音队列("欢迎回家")
  ├── voice_notify == 10 → 高优先级立即播报("已开启预处理")
  └── voice_notify == 11 → 高优先级立即播报("已修改设置")

语音队列:
  高优先级(0x0A/0x0B) → 立即播报不排队
  普通优先级(1/4/9)   → 串行队列FIFO

播报前检查: 模块就绪状态 → 失败重试2次(间隔200ms)
```

### 3. 单流程执行模式（Token 节省）

```
ESP32 上电
  │
  ├── WiFi 连接 → NTP 时间同步
  ├── 传感器采集一次
  ├── HTTP POST /api/run_once { sensors }
  │     │
  │     └── 后端: 采集→AI 决策→执行→返回完整 actions
  │
  ├── 执行 actions (继电器/LED/语音/OLED)
  └── 进入深度休眠 → 等待外部中断唤醒(红外/定时器)
```

## 六大场景链路

### 场景1：语音直控（本地执行）
```
用户说"开灯"
  → 语音模块识别 → UART → ESP32 app_voice.c
  → GPIO10 HIGH(继电器吸合)
  → OLED显示"Light ON"
特点: 零延迟、不联网、不消耗Token
```

### 场景2：环境异常AI询问
```
光线ADC > 3500 且 PIR=有人
  → sensor_task POST /api/report
  → 后端 ask_ai() → 豆包AI分析
  → 返回 voice_notify=1, relay=0
  → ESP32 voice_task 播报"当前光线太暗，是否要为您开灯"
特点: AI只询问不执行，等待用户语音确认
```

### 场景3：回寝检测
```
PIR 首次有人(每日限一次)
  → POST /api/report context.event="human_first"
  → 写入 return_history(date, time, temp, hum, weekday)
  → 返回 voice_notify=9 "欢迎回家"
  → AI 根据环境决策是否开灯/空调
特点: 记录真实回寝数据用于预测
```

### 场景4：预处理预测回寝
```
predict_task 每分钟检查
  → POST /api/report context.query_type="predict"
  → 后端读取 preferences.custom_return_time
  → 当前时间距设定时间≤15分钟 → 豆包AI返回 preheat=1
  → ESP32 开灯(relay=1) + 播报 voice_notify=10
特点: 每天只触发一次(preheat_done_today标记)
```

### 场景5：手机远程控制
```
前端切开关
  → POST /api/control { relay:1 }
  → 后端 pending_control["relay"] = 1
  → ESP32 下次 sensor_task 上报时带走执行
特点: 异步执行，不阻塞前端
```

### 场景6：数据看板
```
前端 GET /api/current → 实时传感器
前端 GET /api/predict/status → 预测状态
前端 GET /api/history/return → 回寝历史
特点: 前端30秒自动刷新，实时时钟前端本地计时
```

## 数据库表设计

### return_history（回寝历史）
| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER | 主键 |
| device_id | TEXT | 设备ID |
| date | TEXT | 日期 YYYY-MM-DD |
| first_seen_time | TEXT | 首次检测到人的时间 |
| weekday | INTEGER | 星期(0-6) |
| temp_at_arrival | REAL | 回寝时温度 |
| hum_at_arrival | REAL | 回寝时湿度 |

### preferences（用户偏好）
| 字段 | 类型 | 说明 |
|------|------|------|
| device_id | TEXT UNIQUE | 设备ID |
| pref_temp | INTEGER | 目标温度(默认24) |
| preheat_enable | INTEGER | 预处理开关(1/0) |
| custom_return_time | TEXT | 手动设定回寝时间 HH:MM |
| pending_notify | INTEGER | 待语音通知标记 |

### predictions（预测记录）
| 字段 | 类型 | 说明 |
|------|------|------|
| device_id | TEXT | 设备ID |
| predict_date | TEXT | 预测日期 |
| predict_time | TEXT | 预测回寝时间 |
| preheat_triggered | INTEGER | 是否已触发 |
| reason | TEXT | 触发原因 |

## AI 决策引擎

### 普通模式（环境异常检测）
- 光线ADC > 3500 且有人 → voice_notify=1（询问开灯）
- 温度 > 30℃ 且有人 → voice_notify=4（询问开空调）
- 温度 < 18℃ 且有人 → voice_notify=4
- 其他 → 不操作

### 预测模式（回寝时间预测）
- 用户手动设定回寝时间 + 当前距设定 ≤ 15分钟 → preheat=1
- 历史数据 ≥ 3天 + 当前距平均 ≤ 15分钟 → preheat=1
- 每天仅触发一次（preheat_done_today 标记）

### 本地 Fallback
AI 调用失败时，后端自动降级到本地时间判断逻辑，确保基本预处理功能不中断。
